## System Architecture

Reelevant is the only **content-first personalisation engine**. Build Content once — the platform creates millions of individualised versions at the moment of interaction, not at send time. The architecture below shows how a single image request triggers real-time personalisation.

```
┌─────────────────────────────────────────────────────────────┐
│                        EMAIL CLIENT                          │
│  Customer opens email → browser requests Reelevant image    │
└──────────────────────────────┬──────────────────────────────┘
                               │ HTTP GET (image request)
                               ▼
┌─────────────────────────────────────────────────────────────┐
│                    REELEVANT EDGE (CDN)                      │
│  Route to nearest region, TLS termination                   │
└──────────────────────────────┬──────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────┐
│                 CONTENT DECISIONING ENGINE                   │
│                                                             │
│  1. Identify customer (ID from URL params)                  │
│  2. Fetch customer data from Datasources (cached + live)    │
│  3. Execute Workflow logic (Branches, conditions)            │
│  4. Select Output Node → render Content                     │
│  5. Return personalised image/HTML                          │
│                                                             │
│  Availability: 99.9% SLA                                   │
└──────────────────────────────┬──────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────┐
│                      DATA LAYER                              │
│                                                             │
│  ┌─────────────┐  ┌──────────────┐  ┌───────────────────┐  │
│  │ Datasources │  │ Workflow      │  │ Event             │  │
│  │ (CRM, Prod, │  │ Definitions  │  │ Stream            │  │
│  │  Stock...)  │  │ (DAG logic)  │  │ (impressions,     │  │
│  └─────────────┘  └──────────────┘  │  clicks, polls)   │  │
│                                      └───────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

## How It Works — Request Lifecycle

1. **Email send:** Your ESP sends the email with a Reelevant image tag (`<img src="https://reelevant.run/...">`)
2. **Email open:** The email client requests the image from Reelevant's edge
3. **Customer identification:** The URL contains a customer identifier (query parameter) mapped to your Datasource records — treat it as public
4. **Data retrieval:** The engine fetches the customer's current data from connected Datasources (from cache, with fallback to live query)
5. **Workflow execution:** The Workflow DAG evaluates conditions (Branches), retrieves additional data (Data Nodes), and selects the appropriate Output Node
6. **Content rendering:** The selected Content is rendered with individual data (product images, prices, offers, text) and returned as an image or HTML fragment
7. **Event capture:** Impression event is recorded. Click tracking captures interactions for future Workflow logic

### Authentication

**Content integration — no auth**

Runner and integration URLs are publicly accessible. There is no authentication at request time. Customer IDs passed in URL parameters are visible to anyone who can see the URL (email HTML, browser, logs). Use opaque identifiers — never put PII directly in integration URLs.

**Reelevant platform — authenticated access**

The Reelevant application supports:

- **Email and password** — standard user login
- **SSO** — single sign-on via your organisation's identity provider

Authorization is enforced with **RBAC** and **ABAC**:

- **RBAC** — [roles](/product-guide/account/roles) define which actions a user can perform on each resource type
- **ABAC** — **company** and **team** attributes scope access so users only see resources assigned to their teams

See [Permissions](/product-guide/account/permissions) for details.

**Datasources API**

- Bearer token (API key per environment) for data ingestion

## Security & Compliance

- **Public integration URLs:** Customer identifiers in URL parameters are public — use opaque IDs, not PII
- **Data minimisation:** Only fields explicitly mapped in Datasources are accessible
- **Retention controls:** Configurable per Datasource (default 90 days for events)
- **Access control:** RBAC (roles) + ABAC (company and team scoping), with environment separation (staging/production)
- **Audit logging:** All admin actions logged with user attribution
- **Penetration testing:** Annual third-party assessment
