## Overview

Tracking Datasources (website events and purchases) store their events in Reelevant and can also replicate them to a BigQuery table for downstream use (for example, scoring). You configure this replication â€” and the underlying storage options â€” with the `patch` step of the datasources API.

<Warning>
  This is only supported for the two tracking Datasources: **website events** and **purchases**. The API technically accepts the call on any Datasource that exposes `storageOptions`, but replication is only wired up for tracking Datasources, and the destination `dataset` is restricted to your Reelevant tracking events dataset (`tracking_events_production_eu`). Any other `dataset` is rejected.
</Warning>

## Configure destinations

Send a `patch` step to the Datasource. Only the sub-options you include are updated; everything else is left untouched.

<CodeGroup>

```bash cURL
curl -XPOST https://api.reelevant.com/v2/datasources/6718ad69056acb425fbecf1a/steps \
  -H "Authorization: Bearer ${access_token}" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "patch",
    "payload": {
      "storageOptions": {
        "destinations": [
          {
            "type": "bigquery",
            "dataset": "tracking_events_production_eu",
            "tableName": "tracking_events_{COMPANY_ID}",
            "deduplicateOptions": { "enabled": true }
          }
        ]
      }
    }
  }'
```

```typescript TypeScript
const datasourceId = '6718ad69056acb425fbecf1a'

const response = await fetch(
  `https://api.reelevant.com/v2/datasources/${datasourceId}/steps`,
  {
    method: 'POST',
    headers: {
      Authorization: `Bearer ${accessToken}`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      name: 'patch',
      payload: {
        storageOptions: {
          destinations: [
            {
              type: 'bigquery',
              dataset: 'tracking_events_production_eu',
              tableName: 'tracking_events_{COMPANY_ID}',
              deduplicateOptions: { enabled: true },
            },
          ],
        },
      },
    }),
  },
)

if (!response.ok) {
  const error = await response.json()
  throw new Error(`Failed to configure destinations: ${error.message}`)
}
```

</CodeGroup>

A successful call returns the current and next steps:

```json
{
  "status": "success",
  "data": {
    "current": { "name": "patch" },
    "next": []
  }
}
```

## Payload reference

The `payload.storageOptions` object accepts the following sub-options. Every field is optional â€” send only what you want to change.

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `destinations` | `Destination[]` | No | `[]` | BigQuery tables the Datasource replicates to. Replaces the current list. |
| `partitionCount` | `number` | No | `1` | Number of partitions for the underlying storage. Positive integer. |
| `segments.offlineSize` | `number` | No | `390000` | Offline segment size. Positive integer. |
| `segments.realtimeSize` | `number` | No | `390000` | Realtime segment size. Positive integer. |
| `segments.offlineReplication` | `number` | No | `2` | Offline segment replication factor. Positive integer. |
| `segments.realtimeReplication` | `number` | No | `1` | Realtime segment replication factor. Positive integer. |
| `retention.value` | `number` | No | `90` | Retention duration. Positive integer. |
| `retention.unit` | `"DAYS"` | No | `"DAYS"` | Retention unit. Only `"DAYS"` is supported. |

Each entry in `destinations` is a BigQuery table:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `type` | `"bigquery"` | Yes | Destination type. Only `"bigquery"` is supported. |
| `dataset` | `string` | Yes | Target dataset. Must be your tracking events dataset (`tracking_events_production_eu`). |
| `tableName` | `string` | Yes | Target table. May contain the `{COMPANY_ID}` and `{DATASOURCE_ID}` placeholders, which are substituted at write time. Otherwise letters, numbers and underscores only. |
| `deduplicateOptions.enabled` | `boolean` | No | When `true`, a periodic job deduplicates rows in the table (dedup key depends on the table: purchases vs website events). |

<Note>
  `partitionCount`, `segments` and `retention` are applied to the underlying table on the next `validate` step. `destinations` are persisted immediately.
</Note>

## Read the current configuration

Fetch the Datasource steps to read back the stored `storageOptions` from the `patch` step value:

```bash cURL
curl https://api.reelevant.com/v2/datasources/6718ad69056acb425fbecf1a/steps \
  -H "Authorization: Bearer ${access_token}"
```

## Error handling

Invalid storage options are rejected with HTTP `400` and error code `1028` (`InvalidStorageOptions`). The `data.reason` field explains what failed.

```json
{
  "status": "fail",
  "error_code": 1028,
  "message": "The storage options are invalid",
  "data": {
    "reason": "`destinations[0].dataset` must be \"tracking_events_production_eu\""
  }
}
```

Common reasons:

- `dataset` is not the tracking events dataset.
- `tableName` is empty or contains characters other than letters, numbers, underscores and the allowed placeholders.
- `partitionCount`, a `segments.*` value or `retention.value` is not a positive integer.
- `destinations` contains two entries with the same `dataset.tableName`.
